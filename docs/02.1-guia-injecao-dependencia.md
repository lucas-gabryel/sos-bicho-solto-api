# 02.1 — Guia prático: Repository Pattern e Injeção de Dependência

> Este é um **complemento didático** do [02 — Arquitetura](./02-arquitetura.md). O doc 02 mostra o
> "o quê"; aqui explicamos o "porquê" e o "como", peça por peça, porque esse padrão vai se repetir
> em **todos os módulos** (animais, tutores, usuários, adoções...).

## O problema que estamos resolvendo

A forma "ingênua" seria o Service usar o Prisma direto:

```ts
// ❌ jeito acoplado — Service conhece o Prisma
export class AnimaisService {
  constructor(private readonly prisma: PrismaService) {}

  buscarPorId(id: string) {
    return this.prisma.animal.findUnique({ where: { id } });
  }
}
```

Funciona, mas tem 2 problemas:

1. **Teste difícil:** para testar o Service sem banco, eu teria que simular o Prisma inteiro.
2. **Acoplamento:** se um dia trocarmos o Prisma por outra coisa, mexo no Service inteiro.

A solução é o **Repository Pattern + Injeção de Dependência (DI)**: o Service passa a depender de um
**contrato** (interface), e quem decide qual implementação real entregar é o **módulo**. Vamos montar
isso peça por peça.

---

## Peça 1 — A interface (o contrato)

> **Sua dúvida:** "o exemplo de contrato é só dizer que tipo de métodos têm que ser implementados,
> definindo entrada e saída?" → **Exatamente isso.**

```ts
// animais.repository.interface.ts
export interface IAnimaisRepository {
  criar(dados: CriarAnimalData): Promise<Animal>;
  buscarPorId(id: string): Promise<Animal | null>;
  desativar(id: string): Promise<void>; // soft delete (ver doc 04) — NÃO apaga do banco
}
```

Uma interface **só descreve a forma**: o nome de cada método, o que ele **recebe** (parâmetros/entrada)
e o que ele **devolve** (retorno/saída). **Não tem corpo, não tem lógica.** É uma promessa: "quem
implementar isso é obrigado a ter esses métodos, com essas entradas e saídas".

Pense num contrato de aluguel: ele diz _o que_ tem que ser cumprido, mas não _como_ a pessoa vai pagar.

Dois pontos importantes:

- **A interface some quando o código vira JavaScript.** Interface é coisa só do TypeScript, serve para
  o compilador conferir os tipos. No código final (JS) ela **não existe**. Guarde isso — é a razão da
  Peça 2 existir.
- **Por que ajuda nos testes:** como o Service vai depender da interface (e não do Prisma), nos testes
  a gente entrega um objeto "de mentira" (mock) que também cumpre o contrato. Ver [08](./08-testes.md).

A implementação **de verdade** fica em outro arquivo:

```ts
// animais.prisma.repository.ts
@Injectable()
export class AnimaisPrismaRepository implements IAnimaisRepository {
  constructor(private readonly prisma: PrismaService) {}

  criar(dados: CriarAnimalData): Promise<Animal> {
    return this.prisma.animal.create({ data: dados }); // AQUI tem corpo/lógica
  }
  buscarPorId(id: string): Promise<Animal | null> {
    return this.prisma.animal.findUnique({ where: { id } });
  }
  async desativar(id: string): Promise<void> {
    // soft delete: o projeto NUNCA apaga fisicamente (ver doc 04). "Excluir" = ativo: false.
    await this.prisma.animal.update({ where: { id }, data: { ativo: false } });
  }
}
```

Repare no `implements IAnimaisRepository`: é o TypeScript **obrigando** essa classe a cumprir o
contrato. Se faltar um método ou o tipo estiver errado, dá erro de compilação. Esta é a **única**
classe que conhece o Prisma.

> **Soft delete x hard delete:** no nosso projeto "excluir" entidades de domínio (usuário, tutor,
> animal) é **soft delete** (`ativo = false`). `prisma.delete` (apagar de vez) só é usado em casos
> pontuais como remover uma **foto** do animal. Por isso o método aqui se chama `desativar`, não
> `remover`.

## Peça 2 — O `Symbol` (o token / a "etiqueta")

> **Sua dúvida:** "pra que serve o Symbol?"

```ts
export const ANIMAIS_REPOSITORY = Symbol("ANIMAIS_REPOSITORY");
```

Para o NestJS **entregar** uma dependência, ele precisa de uma **chave** (chamamos de _token_) que ligue
"quem precisa" a "o que entregar". É como uma **etiqueta com nome** num guarda-volumes: você entrega a
etiqueta e recebe o objeto certo.

- Quando a dependência é uma **classe**, a própria classe serve de etiqueta (o NestJS usa o nome dela).
- Mas a nossa dependência é uma **interface**, e — como vimos na Peça 1 — **interface não existe no
  código final**. Logo, ela **não pode** ser usada como etiqueta. Precisamos de um valor real, que
  exista em tempo de execução, para servir de chave.

Poderíamos usar uma string (`'ANIMAIS_REPOSITORY'`), mas strings podem **colidir** (dois lugares usando
o mesmo texto sem querer). Um `Symbol` é **garantidamente único** — dois `Symbol('x')` são diferentes
entre si. Por isso ele é a etiqueta perfeita: nunca colide com a de outro módulo.

> Resumo: o `Symbol` é a **etiqueta única** que conecta o pedido do Service (Peça 4) com o registro
> feito no módulo (Peça 3). Ele existe **porque a interface some na compilação**.

---

## Peça 3 — O registro no módulo (juntando as peças)

> **Sua dúvida:** "não entendi nada do registro do módulo." → É aqui que dizemos ao NestJS
> **como montar** cada coisa. Vamos linha por linha.

```ts
@Module({
  controllers: [AnimaisController],
  providers: [
    AnimaisService,
    { provide: ANIMAIS_REPOSITORY, useClass: AnimaisPrismaRepository },
  ],
})
export class AnimaisModule {}
```

O NestJS tem um "montador de objetos" (o **container de DI**). O array `providers` é a **lista de
receitas**: ensina o container a fabricar cada dependência. Há duas receitas aqui:

**Receita 1 — `AnimaisService`** (forma curta)
Escrever só a classe é um atalho que significa:

> "Quando alguém precisar de um `AnimaisService`, crie uma instância de `AnimaisService`."

A etiqueta e a implementação são a mesma classe, então não precisa detalhar.

**Receita 2 — `{ provide: ANIMAIS_REPOSITORY, useClass: AnimaisPrismaRepository }`** (forma completa)
Aqui a etiqueta e a implementação são **diferentes**, então explicamos os dois:

| Parte                               | Significado                                                                              |
| ----------------------------------- | ---------------------------------------------------------------------------------------- |
| `provide: ANIMAIS_REPOSITORY`       | A **etiqueta** (token) que as pessoas vão pedir. É o nosso `Symbol`.                     |
| `useClass: AnimaisPrismaRepository` | A **classe concreta** que o Nest vai instanciar e entregar quando pedirem essa etiqueta. |

Lendo em português:

> "Quando alguém pedir a etiqueta `ANIMAIS_REPOSITORY`, fabrique um `AnimaisPrismaRepository` e
> entregue."

**Por que isso é poderoso (e cai no SOLID):** o Service vai pedir a _etiqueta_ (algo abstrato), e é o
**módulo** que decide qual classe concreta cumpre o papel. Se um dia você quiser trocar a implementação
— por exemplo, usar um repositório falso, ou outro banco — você muda **só esta linha**:

```ts
// trocar a implementação sem tocar no Service:
{ provide: ANIMAIS_REPOSITORY, useClass: AnimaisEmMemoriaRepository }
```

O `AnimaisService` continua **idêntico**. Isso é o "D" (Dependency Inversion) e o "O" (Open/Closed)
do SOLID na prática.

> ⚠️ Para o `AnimaisPrismaRepository` conseguir receber o `PrismaService` no construtor dele, o
> `PrismaModule` precisa estar disponível (normalmente importado de forma global). Ver
> [03 — Estrutura de Pastas](./03-estrutura-de-pastas.md).

## Peça 4 — O Service recebendo a dependência (constructor + `@Inject`)

> **Suas dúvidas:** "no service você tá pegando a interface e fazendo o quê? por que dentro de um
> constructor? por que `@Inject`?"

```ts
@Injectable()
export class AnimaisService {
  constructor(
    @Inject(ANIMAIS_REPOSITORY)
    private readonly animaisRepository: IAnimaisRepository,
  ) {}

  buscarPorId(id: string) {
    return this.animaisRepository.buscarPorId(id); // usa o repositório aqui
  }
}
```

### Por que no construtor?

O Service **não cria** suas dependências (ele nunca faz `new AnimaisPrismaRepository()`). Em vez disso,
ele **declara no construtor**: "para eu existir, preciso receber algo que cumpra `IAnimaisRepository`".
Quando o NestJS fabrica o `AnimaisService`, ele olha o construtor, vê o que falta e **entrega pronto**.
Isso se chama **Inversão de Controle**: quem manda buscar as peças é o framework, não o seu código.

> Analogia: você não fabrica o motor do seu carro; ele já vem instalado. Você só usa.

### O que o `private readonly` faz?

Esse atalho do TypeScript faz **duas** coisas de uma vez:

1. Guarda o que foi recebido numa propriedade da classe (`this.animaisRepository`), para você usar nos
   métodos (como no `buscarPorId`).
2. `private` = só dá pra usar **dentro** do Service. `readonly` = ninguém reatribui depois (segurança).

Ou seja: "pegar a interface" significa **guardar a dependência recebida** para chamar seus métodos
(`this.animaisRepository.criar(...)`, etc.). Note que o tipo é a **interface** — o Service nem sabe que
por baixo é Prisma.

### Por que precisa do `@Inject(...)` aqui?

Normalmente o NestJS descobre o que injetar **pelo tipo**. Ex.: se você escreve
`private prisma: PrismaService`, ele vê o tipo `PrismaService` (uma classe, que existe em runtime) e
sabe o que entregar — **sem** precisar de `@Inject`.

Mas o nosso tipo é `IAnimaisRepository`, uma **interface** — e (de novo) interface **some na
compilação**. O NestJS olha o construtor e não acha tipo nenhum para usar de etiqueta. Então a gente
**aponta explicitamente** a etiqueta certa:

```ts
@Inject(ANIMAIS_REPOSITORY)  // "injete o que estiver registrado sob esta etiqueta"
```

É isso que fecha o circuito: `@Inject(ANIMAIS_REPOSITORY)` (Peça 4) bate com
`provide: ANIMAIS_REPOSITORY` (Peça 3), que aponta para `AnimaisPrismaRepository` (Peça 1).

> **Regra de bolso:** injetou uma **classe**? Não precisa de `@Inject`. Injetou via **interface/token
> (Symbol)**? **Precisa** de `@Inject(token)`.

## Como tudo se conecta (mapa mental)

```
   Service pede                 Módulo decide               Nest entrega
┌──────────────────┐         ┌──────────────────┐        ┌─────────────────────────┐
│ @Inject(         │         │ provide:         │        │ new AnimaisPrismaRepo... │
│   ANIMAIS_REPO ) │ ──etiqueta──▶ ANIMAIS_REPO  │ ─usa──▶ │ (que implementa a        │
│ repo: IAnimais...│         │ useClass: ...Repo│        │  interface)              │
└──────────────────┘         └──────────────────┘        └─────────────────────────┘
        ▲                                                              │
        └──────────── o Service usa via this.animaisRepository ◀───────┘
```

A mesma `Symbol` (`ANIMAIS_REPOSITORY`) aparece nos **dois lados** (quem pede e quem registra) — é o
fio que liga tudo.

## Receita: criando isso para um módulo novo (passo a passo)

Sempre que criar um módulo novo (ex.: `tutores`), repita estes 5 passos:

1. **Criar a interface** `tutores.repository.interface.ts` com os métodos (entradas/saídas).
2. **Criar o token** no mesmo arquivo: `export const TUTORES_REPOSITORY = Symbol('TUTORES_REPOSITORY');`
3. **Criar a implementação** `tutores.prisma.repository.ts` com `@Injectable()` e
   `implements ITutoresRepository`, usando o Prisma de verdade.
4. **Registrar no módulo:**
   ```ts
   providers: [
     TutoresService,
     { provide: TUTORES_REPOSITORY, useClass: TutoresPrismaRepository },
   ];
   ```
5. **Injetar no Service:**
   ```ts
   constructor(
     @Inject(TUTORES_REPOSITORY)
     private readonly tutoresRepository: ITutoresRepository,
   ) {}
   ```

Pronto: trocou os nomes, manteve o padrão. Todo módulo fica igual — fácil de revisar e dividir.

## Bônus — "vazamento de stack trace e query em produção"

> **Sua dúvida:** "o que seria esse vazamento de dados (stack trace e query) em produção?"

Quando um erro acontece, o Node gera um **stack trace**: um relatório detalhado mostrando a sequência de
funções, **arquivos e linhas** onde o erro ocorreu. É ouro para você depurar — mas é **perigoso** se for
devolvido na resposta da API para o usuário/atacante, porque revela a estrutura interna do sistema
(caminhos de arquivo, bibliotecas, organização do código).

A **query** é o comando SQL que o Prisma manda ao banco. Se um erro de banco vier "cru" na resposta,
ele pode expor **nomes de tabelas, colunas e até dados** — um prato cheio para quem quer atacar.

**Exemplo do que NÃO pode vazar** (resposta de erro ruim):

```json
{
  "message": "PrismaClientKnownRequestError: Unique constraint failed",
  "stack": "at AnimaisPrismaRepository.criar (/app/src/modules/animais/...:42:15)\n at ...",
  "query": "INSERT INTO animais (email, cpf) VALUES ($1, $2)"
}
```

**O que a API deve devolver** (limpo e genérico):

```json
{ "statusCode": 500, "message": "Erro interno do servidor" }
```

Os detalhes completos (stack, query) vão **só para o log do servidor**, onde só a equipe vê. Quem cuida
disso é o **Exception Filter global** (ver [02](./02-arquitetura.md) e [07](./07-padroes-e-convencoes.md)):
em produção ele esconde os detalhes; em desenvolvimento pode mostrar mais para facilitar o debug.

> Regra: **erro detalhado para o log; mensagem genérica para o cliente.**
